How MyHerder Works

A plain-language guide to your app — no programming experience required.

What is MyHerder?

MyHerder is a web app for managing a dairy farm. Farmers use it on their phones and computers to:

  • Keep track of their cows (names, tags, status, breed)
  • Record milk production for each cow, each milking session
  • Log health issues (e.g. mastitis, lameness) and track treatments
  • Manage breeding events (insemination, pregnancy checks, calving)
  • View analytics — charts and reports about herd performance
  • Download PDF/Excel reports for record-keeping
  • Work offline — data saves on the phone and syncs when internet returns
  • Support multiple farms — each farm's data is completely separate

It supports two languages: English and Afrikaans.

The Frontend — What Users See

The frontend is everything that runs on the user's phone or computer. It's the app itself — the buttons, lists, forms, and charts.

Think of it as the dining room of the restaurant. The customer (user) interacts with the menu, plates, and decor. They never see the kitchen. The frontend is built entirely with Vue 3.

How a page is built

Every page you see is a View — a file that describes what should appear on screen. For example:

  • CowListView — the page showing all your cows in a scrollable list
  • CowDetailView — the page showing one cow's full details
  • MilkRecordingView — the page where you log milk for each cow
  • DashboardView — the home page with summary stats

Each view is made of smaller reusable pieces called components, organized by size:

Atoms Tiny single-purpose pieces.
Example: SyncIndicator (the online/offline dot)
Molecules Small combos of atoms.
Example: ConfirmDialog ("Are you sure?" popup), SearchInput
Organisms Bigger self-contained sections.
Example: AppHeader (the top bar), BottomNav, CowCard
Views Full pages — made from organisms, molecules, and atoms.
30+ views in the app.
Restaurant analogy: An atom is a single napkin. A molecule is a place setting (napkin + fork + knife). An organism is a fully set table. A view is the entire dining room layout.

The Backend — The Engine

The backend is a program running on a server computer (in our case, a web hosting service called cPanel). Users never see it directly. It:

  • Receives requests from the app ("give me all cows", "save this milk record")
  • Checks security — are you logged in? Do you have permission?
  • Validates data — is the milk amount a real number? Is the tag not empty?
  • Reads/writes the database — fetches or stores the actual data
  • Sends responses — returns the requested data (or an error message)
Think of it as the kitchen. Orders come in, they're validated, ingredients are fetched from the pantry, food is prepared, and plates go back out. The customer never sees any of this.

The API — The menu of what the server can do

The backend offers a list of "endpoints" — specific URLs that do specific things. Think of them as items on a menu. Here are some examples:

Migrations — Building the filing cabinet

When the app needs a new table or a new column, we write a migration — a small instruction file that says "create this table" or "add this column". We have 34 of these, numbered 001 to 034. They run in order, building the database structure step by step.

Analogy: Migrations are like assembly instructions for a filing cabinet. Step 1: build the main frame (cows table). Step 15: add the breeding drawer. Step 30: add the farms label system. You run them in order, and at the end, your filing cabinet is fully built.

Logging In

There are two ways to log in, depending on your role:

Admin Login (Password)

Type your username and password. Optionally enter a farm code if managing multiple farms. Your session lasts 24 hours. Can optionally use 2FA (a code from an authenticator app).

Worker Login (PIN)

Type your username, 4-digit PIN, and your farm code (required). Simpler login for farm workers. Your session lasts 7 days so you don't have to log in every day.

The ID Badge (JWT Token)

When you log in successfully, the server creates a digital ID badge for you. In technical terms, it's called a JWT (JSON Web Token). Here's what's on it:

Your Digital ID Badge Who you are: id, username, full_name
Your role: admin, worker, or super_admin
Your permissions: can_record_milk, can_log_breeding, etc.
Your farm: farm_id (which farm you belong to)
Your language: en or af
Expiry time: when this badge expires
Version number: so the admin can revoke your access if needed

This badge is saved on your phone in two places (for safety). Every time the app talks to the server, the badge is sent along so the server knows who's asking. It's like showing your wristband to every staff member.

Auto-refresh: When your badge is about to expire (less than 1 hour left), the app quietly asks the server for a fresh one — so you don't get randomly logged out.

Revocation: If an admin revokes your access, they bump a version number in the database. The next time your app contacts the server, it compares version numbers, sees they don't match, and forces you to log out.

There are three roles:

Admin A farm manager Everything within their own farm. Can manage users, settings, see reports. Automatically has all permissions.

"Read mirrors write" — if you don't have permission to record milk, you can't even see milk records. The same permission gates both reading and writing. This keeps the app simple for workers — they only see features relevant to their job.

Multi-Farm — Invisible Walls Between Farms

MyHerder supports multiple farms. Each farm's data is completely separate — Farm A can never see Farm B's cows, milk records, or users. This is called multi-tenancy.

Analogy: an apartment building. Each apartment (farm) has its own locked door. Tenants can only access their own apartment. The building manager (super-admin) has a master key that works on all doors.

How the walls work

1. Your ID badge says which farm you belong to When you log in, the JWT token includes your farm_id
▼
2. Every request is tagged with your farm The Tenant Scope bouncer reads your farm_id and stamps it on the request
▼
3. Every database query filters by farm "Give me cows WHERE farm_id = this person's farm" — other farms' cows are simply invisible

This applies to everything — cows, milk records, breeding events, health issues, users, settings. No exceptions.

Even the phone's local storage is separated: each farm gets its own database named myherder_db_[farmId]. If a user switches farms, a completely different local database opens.

The Master Key (Super-Admin)

As the super-admin, you can see across all farms. But you can also "enter" a specific farm to manage it as if you were its admin.

Logged in as Super-Admin No farm context — you see the global dashboard: farm list, defaults, announcements
▼ click "Enter Farm" on a specific farm
Server issues a temporary badge A special 4-hour ID badge with that farm's ID on it. Your original badge is saved for later.
▼
You're now "inside" the farm You see the same interface as a farm admin. All data is scoped to this farm. Phone opens this farm's local storage.
▼ click "Exit Farm" in your profile
Original badge restored Back to the global super-admin view.

Working Without Internet

Farm workers are often in areas with bad internet. MyHerder handles this:

The app loads without internet

A Service Worker (background helper) caches the entire app on your phone. So even offline, you can open it and navigate around.

Data is cached on your phone

Every time data loads from the server, a copy is saved locally (in IndexedDB via Dexie). If the server is unreachable, the app shows the last cached version.

Changes are queued

If you create a cow or log milk while offline, it's saved locally and added to a "sync queue" — a list of changes waiting to be sent to the server.

The Outbox (Sync Queue)

Analogy: writing letters when the postman can't come. You write your letters (create data) and put them in the outbox (sync queue). When the postman arrives (internet returns), he picks up all the letters and delivers them. If a letter conflicts with something at the destination (someone else changed the same thing), the most recent change wins.
When offline:
1. User acts Records 5 litres of milk for cow #42
2. Saved locally Stored in phone's IndexedDB. User sees it immediately.
3. Added to outbox sync queue: "create milk record, 5L, cow #42"
▼ internet comes back
When back online:
4. Background push Service worker sends all queued changes to server
5. Server processes Saves each change. If conflict, latest timestamp wins.
6. Queue cleared Local data replaced with server's version. Outbox emptied.

On/Off Switches (Feature Flags)

Admins can turn entire features on or off per farm. This is useful when a farm doesn't need breeding tracking, for example.
